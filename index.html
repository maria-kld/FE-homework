Основні команди Git

    Перелік основних команд:

  • git config --global user.name "my_name"

    • Ім'я користувача потрібно, щоб прив'язувати коміти до вашого імені. Це не те саме, що ім'я користувача облікового запису GitHub, за допомогою якого виконується вхід у профіль GitHub. Ви можете задати або змінити ім'я користувача за допомогою команди git config. Нове ім'я автоматично відображатиметься в наступних комітах, відправлених на GitHub через командний рядок. Якщо хочете приховати своє реальне ім'я, можна використовувати ім'я користувача Git довільний набір символів.

  • git config --global user.email "my_email"

    • Командою git config можна змінювати адресу електронної пошти, прив'язану до ваших комітів Git. Нова адреса електронної пошти автоматично відображатиметься у всіх подальших комітах, поданих на GitHub через командний рядок

  • get init

    • Створити порожній репозиторій Git або знову ініціалізувати існуючий можна init. При ініціалізації він створить приховану папку. У ній містяться всі об'єкти та посилання, які Git використовує та створює в історії роботи над проектом.

  • git status

    • Переглянути статус потрібного репозиторію можна за ключовим словом status: його дія поширюється на підготовлені, непідготовлені та файли, що не відстежуються.

  • git add .

    • Добавляє всі файли і папки

  • git commit -m

    • Під час створення комміту в репозиторії можна додати однорядкове повідомлення за допомогою параметра commit із прапором -m. Саме повідомлення вводиться безпосередньо після прапора, у лапках.

  • git commit

    • Також можна відкрити текстовий редактор у терміналі для написання повного повідомлення комміту. Воно може складатися з декількох рядків тексту, в якому характеризуються докладно зміни, внесені в репозиторій.

  • git log

    • Переглядати зміни, внесені до репозиторію, можна за допомогою параметра log. Він відображає список останніх коммітів у порядку виконання.

  • git checkout

    • Відновити файли робочого дерева, які не підготовлені до комміту, можна параметром checkout. Для операції потрібно вказати шлях до файлу. Якщо шлях не вказано, параметр git checkout змінить вказівник HEAD, щоб вказати вказану гілку як поточну.

  • git branch

    • Створити нову гілку можна за допомогою branch, вказавши ім'я гілки.

  • git puch

    • Відправляти зміни у віддалений репозиторій можна параметром push із зазначенням імені репозиторію та гілки.

  • git pull

    • Для завантаження змін віддаленого репозиторію використовується параметр pull. Він завантажує копію поточної гілки із зазначеного віддаленого репозиторію та поєднує її з локальною копією.

  • git merge

    • Об'єднати дві гілки можна параметром merge із зазначенням імені гілки. Команда об'єднає вказану гілку з основною.

  • git clone

    • Сама команда має таку форму записи:
    git clone <силка на репозиторий > <назва папки >
    Тобто ми просимо Git створити копію репозиторію, який знаходиться за посиланням (<посилання на репозиторій>), і можемо вказати назву нової папки, в яку Git скопіює репозиторій (<назва папки>). Якщо його не вказати, папка називатиметься так само, як і сам репозиторій.


Що таке файл .gitignore?

Файл Git Ignore визначає, які файли та папки не слід ігнорувати у даному вихідному коді. Його можна використовувати на основі кожного шляху, щоб правила застосовувались лише до певних папок, але ви також можете створити глобальний файл GITIGNORE, який застосовується до кожного сховища Git, яке у вас є. Ви можете знайти десятки прикладів файлів GITIGNORE, які рекомендуються в різних сценаріях, на сторінці шаблонів .gitignore GitHub.


    Advanced

  • git revert

    • Команда git revert – найбезпечніший спосіб скасувати операцію без втрати історії коммітів. Команда скасовує дії минулих коммітів, створюючи новий, що містить всі скасовані зміни. Ця команда корисна, коли ви вже запушили зміни у віддалений репозиторій, оскільки вона зберігає незайманим вихідний коміт.

  • git reset

    • Команда git reset, як можна здогадатися із назви, використовується в основному для скасування змін. Вона змінює вказівник HEAD і, опціонально, стан індексу. Також ця команда може змінити файли в робочому каталозі при використанні параметра --hard, що може призвести до втрати напрацювань при неправильному використанні, тому переконайтеся в серйозності своїх намірів перш ніж використовувати його.

  • git rebase

    • ще один спосіб перенести зміни з однієї гілки до іншої. Rebase стискає всі зміни в один патч. Потім він інтегрує патч у цільову гілку. На відміну від злиття, переміщення перезаписує історію, тому що вона передає завершену роботу з однієї гілки до іншої.

  • gir cherry-pick

    • Вона бере зміни, що вносяться одним коммітом, і намагається повторно застосувати їх у вигляді нового комміту в поточній гілці. Ця можливість корисна в ситуації, коли потрібно забрати парочку коммітів з іншої гілки, а не зливати гілку цілком з усіма змінами, що внесені до неї.

  • git stash

    • Вона призначена для того, щоб помістити поточні зміни, які ви здійснили у файлах, в окреме сховище, та повернути файли до вихідного стану. Тобто git stash ховає зміни у файлах та зберігає ці зміни окремо, щоб потім можна було їх повернути
